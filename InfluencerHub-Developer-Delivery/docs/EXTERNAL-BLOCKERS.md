# External Blockers — InfluencerHub

توثيق صادق للأعمال المعتمدة على اعتمادات/قرارات خارجية لا يمكن للبناء الذاتي إكمالها. لكل بند: العقد (Contract) + المُهايئ (Adapter) + وضع الاختبار (Sandbox/Mock) + راية الميزة (FeatureFlag) + الاختبارات + الحالة الصادقة. **لا محاكاة تسليم زائفة**؛ الحالات الصريحة: `WAITING_FOR_CREDENTIALS` / `SANDBOX` / `MOCK` / `DISCONNECTED` / `ERROR`.

> القاعدة: نبني الطبقة المحايدة للمزوّد (عقد + مُهايئ + fake) ونتركها بحالة صادقة حتى تُوفَّر الاعتمادات/القرارات الخارجية. لا ندّعي «جاهز للإنتاج» بلا دليل حقيقي.

## المُنفَّذ فعليًا بهذا النمط (ليس فراغًا)
- **OTP (Phase 4)**: `OtpMailer` (بريد فعلي في الإنتاج / تسجيل محليًا) + `OtpSmsSender` = `NullSmsSender` بحالة `waiting_for_credentials` (لا يُظهر الرمز في الإنتاج). ✅ مبني ومختبَر.
- **الإشعارات (Phase 5)**: `NotificationService` — in_app فعلي؛ email/sms تُسجَّل كمحاولة `waiting_for_credentials` (لا تسليم وهمي). ✅ مبني ومختبَر.
- **المستحقات (Phase 11)**: النظام **لا ينفّذ أي تحويل مالي**؛ يتتبّع الالتزامات بحالات صادقة (`waiting_for_provider`)؛ «مدفوع» تُسجَّل يدويًا بمرجع بعد تسوية حقيقية عبر بنك/مزوّد خارجي. ✅ مبني ومختبَر.

## الأعمال المؤجّلة على اعتمادات خارجية (Phases لاحقة)

### 1) الإعلانات المدفوعة (Paid Media)
- **البلوكر**: اعتمادات منصّات الإعلانات (Meta Ads / TikTok Ads / Google Ads / Snap Ads) — عقود API + رموز وصول لكل عميل.
- **الخطة**: عقد `AdPlatformClient` (fetchCampaigns/spend/metrics) + مُهايئات لكل منصّة + `FakeAdPlatformClient` للاختبار + راية `paid_media.enabled`. الحالة الافتراضية `WAITING_FOR_CREDENTIALS`.
- **قرار المستخدم المطلوب**: أي منصّات، وحسابات الإعلانات، وربط OAuth لكل عميل.

### 1ب) الاكتشاف الحيّ للناشرين (Publisher Intelligence — Live Discovery)
- **البلوكر**: اعتمادات APIs الرسمية للاكتشاف/التحليل لكل منصّة (Snapchat / TikTok / Instagram-Meta / YouTube / X / LinkedIn) — رموز وصول + موافقات المنصّات.
- **المُنفَّذ الآن (صادق)**: وحدة الناشرون `/beta/publishers` كاملة (بحث/تحليل/مقارنة/حفظ/تحويل إلى مؤثر idempotent/ربط بترشيح)، مبنية فوق `PlatformRegistry` عبر `PublisherConnectors`. كل ناشر يحمل `source` (manual/import/sandbox/live) و`last_synced_at`. بيانات العرض `source='sandbox'` موسومة بصدق.
- **الحالة**: الاكتشاف الحيّ = `WAITING_FOR_CREDENTIALS` لكل منصّة (كلها available_manual في السجل)؛ اليدوي/الاستيراد متاحان الآن. تُرفَّع الحالة تلقائيًا عند توفّر الاعتماد. **لا أرقام أو تحليلات وهمية**.
- **قرار المستخدم المطلوب**: أي منصّات أولًا + اعتماداتها + نطاق الوصول المسموح.

### 2) مركز التكاملات (Integration Hub)
- **البلوكر**: اعتمادات كل تكامل (منصّات التواصل لجلب المقاييس، بوّابات الدفع، أدوات التحليل).
- **الخطة**: سجلّ موصّلات (`IntegrationConnector`) + حالة اتصال صريحة لكل موصّل (`DISCONNECTED`/`SANDBOX`/`CONNECTED`) + fakes. لا سحب بيانات وهمية.
- **قرار المستخدم المطلوب**: أي تكاملات أولًا + اعتماداتها.

### 2ب) الفوترة التجارية والمدفوعات — ⛔ غير مبنيّة (اكتُشفت في تدقيق الجاهزية)
- **الحالة الصادقة**: النظام يدير الخطط والاشتراكات والحقوق والاستهلاك وحدودها فعليًا ومُختبَرة، لكن **لا يوجد**: مزوّد دفع حقيقي، ولا فواتير (`Invoice`)، ولا مدفوعات (`Payment`)، ولا دفتر أستاذ (`Ledger`)، ولا webhooks فوترة. `FakeBillingProvider` هو المُنفِّذ الوحيد لعقد `BillingProvider` واسمه يقول ذلك.
- **الأثر**: لا يجوز إطلاق فوترة تجارية بهذه الحالة. لا شاشة تدّعي دفعًا ناجحًا (اسم المزوّد يُعرض «بيانات عرض (تجريبي)»).
- **قرار تجاري مطلوب**: المزوّد (ميسر/Stripe) + حساب تاجر + مفاتيح + سياسة ضريبة القيمة المضافة والفوترة الإلكترونية (ZATCA/فاتورة) + دورة الفوترة والاسترداد.

### 2ج) Google Drive — ⛔ غير مبنيّ إطلاقًا
- **الحالة الصادقة**: لا كود ولا إعداد ولا حتى مدخل في سجلّ المنصّات. التخزين كلّه على قرص `local` خاص (لا رابط عام ثابت).
- **قرار المستخدم المطلوب**: هل يُراد Drive أصلًا (مزامنة مستندات؟ تسليم محتوى؟) أم يكفي التخزين الداخلي. عند الرغبة: مشروع Google Cloud + OAuth + نطاق صلاحيات + سياسة خصوصية للملفات المُرسَلة.

### 2د) التحقّق بخطوتين (2FA) — ⛔ ناقص
- **الحالة الصادقة**: الحقل `two_factor_confirmed_at` موجود وتُعرض حالته في صفحة الأمان، لكن **تدفّق التفعيل غير مبنيّ** (لا QR ولا رموز احتياطية ولا تحقّق).
- **قرار منتَج مطلوب**: TOTP أم رسالة نصية (الأخيرة محجوبة بعائق SMS نفسه)، وهل يكون إلزاميًا لبعض الأدوار.

### 3) طبقة الذكاء الاصطناعي (AI Layer)
- **البلوكر**: اعتماد مزوّد نماذج (مفتاح API) وقرار المزوّد.
- **الخطة**: عقد `AiProvider` محايد (اقتراح تسميات/ملخّصات/مطابقة مبدعين) + `FakeAiProvider` حتمي للاختبار + راية `ai.enabled`. الحالة `WAITING_FOR_CREDENTIALS`. (المطابقة الحالية في Phase 8 خوارزمية شفّافة بلا AI — تعمل الآن.)
- **قرار المستخدم المطلوب**: المزوّد + المفتاح + سياسة الخصوصية للبيانات المُرسَلة.

### 4) Staging / النشر
- **البلوكر**: VPS/استضافة + DNS + شهادات + أسرار الإنتاج (يتطلب sudo/وصولًا خارجيًا).
- **الخطة**: توثيق خطوات النشر (env، migrate، queue worker، scheduler) دون تنفيذ فعلي حتى تتوفّر البنية.
- **قرار المستخدم المطلوب**: المزوّد + النطاق + إدارة الأسرار.

### 5) جاهزية الإنتاج
- تعتمد على ما سبق + قرارات قانونية/مالية (مزوّد الدفع الفعلي، اتفاقيات المعالجة، الامتثال).

## الأعمال غير المحجوبة المتبقية (قابلة للبناء الذاتي)
- **محرّك الأتمتة/SLA**: تصعيد تلقائي للطلبات المتأخّرة + تذكيرات SLA (يعتمد على `service_requests.due_at` الموجود) — قابل للبناء بالكامل داخليًا.
- **استيراد Legacy**: أوامر استيراد إضافية (يوجد `import:legacy-clients`).
- **تصلّب أمني**: مراجعة صلاحيات/حدود معدّل/رؤوس أمان.
- **توسيع E2E**: سيناريوهات Playwright لدورة الحياة الكاملة عبر البوابات الأربع.

تُوسَم كل نتيجة بمصدرها ولا تُعرض كأنها مُزامَنة.
