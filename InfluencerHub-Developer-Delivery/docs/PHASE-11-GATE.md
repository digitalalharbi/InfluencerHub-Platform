# Phase 11 — المستحقات المالية (Finance / Payouts) — بوابة القبول

الحالة: **PHASE COMPLETE** (Feature Complete + مُتحقَّق بالمتصفّح + اختبارات خلفية). التالي: Phase 12.

> **صدق مالي**: النظام **لا ينفّذ أي تحويل مالي**. يتتبّع التزامات المستحقات بحالات صادقة؛ التسوية الحقيقية تتم عبر مزوّد/بنك خارج النظام، ويُسجّل الموظّف «مدفوع» يدويًا **بمرجع تحويل** بعد إتمامها. لا حالة «مدفوع» تلقائية ولا تسليم وهمي.

## ما أُنجز
- **البنية** `app/Domain/Finance`: `payouts` + `payout_status_history`؛ `PayoutWorkflowService` (حالات صادقة بالأحداث).
- **الحالات**: pending→approved→scheduled→waiting_for_provider→paid/failed؛ failed→scheduled؛ +cancel. المبلغ بوحدات صغرى؛ لقطة IBAN last4 من الملف المشفّر عند الإنشاء.
- **الوكالة/المالية** `/app/payouts`: إنشاء (لمبدع)، اعتماد، جدولة، إرسال للمزوّد (waiting_for_provider مع تنويه صريح)، **تسجيل الدفع بمرجع إلزامي**، تسجيل فشل بسبب، إلغاء + ملخّص المستحق المفتوح. الصلاحية لأدوار MANAGE_DOCS (تشمل finance).
- **المبدع** `/creator/payouts` (عرض فقط): ملخّص مدفوع/قيد التحصيل + قائمة بحالات مبسّطة + تنويه أن التحويل لحسابه (IBAN) المسجَّل.
- **إشعارات**: جدولة/دفع/فشل → المبدع.

## الأمان (اختبارات + حيًّا)
- عزل مستأجر + IDOR-safe؛ المبدع يرى مستحقاته فقط.
- لا «مدفوع» قبل مرحلة المزوّد؛ آلة حالة صارمة؛ المرجع إلزامي عند الدفع.
- إدارة المستحقات لأدوار مالية/إدارية فقط (viewer ممنوع 403).

## الاختبارات
- **خلفي: 331 إجمالًا**. `PayoutTest(10)`: لقطة IBAN+رقم، رفض مبلغ غير موجب، مسار كامل حتى paid بمرجع، منع paid قبل المزوّد، إعادة جدولة الفاشل، إشعار الدفع، إنشاء+دفع HTTP، رؤية المبدع، منع viewer، عزل مستأجر.
- **حيًّا**: PY-1-1 (Renad، 13,500، IBAN ••••4521) عبر approve→schedule→waiting_for_provider (تنويه «النظام لا ينفّذ التحويل») → تسجيل الدفع بمرجع TRX-… → «مدفوع» + سجل حالة كامل.
