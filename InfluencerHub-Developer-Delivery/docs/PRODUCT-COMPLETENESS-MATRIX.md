# Product Completeness Matrix — InfluencerHub (خريطة تنفيذ حيّة)

خريطة كل الوحدات/الصفحات/البوابات وحالتها. حدّثها بعد كل صفحة. الحالات: ✅ مكتمل React (اجتاز البوابة) · 🟦 Blade فقط (منتج تشغيلي، غير محوّل بعد) · 🟨 قيد التنفيذ · ⬜ غير مبدوء.
مبدأ التحويل: `/beta` React بالتوازي مع Blade `/app`، ثم قص المسار بعد اجتياز البوابة.

## أولوية التنفيذ (React)
DS/Shell ✅ → Dashboards ✅ → المؤثرون ✅ → الحملات ✅ → **العملاء/العلامات (جارٍ)** → الطلبات → الترشيحات → المحتوى → العقود → المالية → التقارير → التكاملات → الإعدادات/الاشتراكات → البوابات → مراجعة شاملة+توحيد.

**نمط الإجراءات مثبت (Requests):** أي إجراء يُغيّر الحالة = مسار `/beta/.../{action}` POST في Inertia controller يعيد استخدام الـWorkflow Service الحالي ويرجع `back()`؛ صفحة React تستدعيه عبر `router.post` (Inertia يتكفّل بـXSRF)؛ الأسباب عبر Modal React (لا window.*). طبّقه على المحتوى/العقود/المستحقات.
**تمّ:** المحتوى/العقود/المالية/التعاونات/التقارير/التكاملات/الإعدادات-الاشتراكات/**الترشيح**. **الخطوة التالية:** **البوابات الخارجية** (العميل/المؤثر/UGC/الطلبات العامة/الشريك/الاشتراك/مدير SaaS) كلٌّ منتج كامل، ثم اختبارات E2E للأدوار/الصلاحيات، ثم **قص `/beta`→`/app`** وإزالة توائم Blade بعد اجتياز البوابة، ثم مراجعة نهائية+توحيد.
**ملاحظة:** الـCRM الأساسي (لوحة/مؤثرون/حملات/عملاء — list+detail) مكتمل في React على `/beta`؛ Blade `/app` باقٍ بالتوازي حتى القص.
