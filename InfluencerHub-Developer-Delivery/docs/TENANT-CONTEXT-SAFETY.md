# سلامة سياق المستأجر

> لماذا هذه الوثيقة: نمط واحد تكرّر **تسع مرّات** في هذا المستودع، وكلّفنا في
> كل مرّة تشخيصًا من الصفر. آخر ظهور له كان أخطرها: وسيط بوابة يهدم السياق
> الذي بناه، فوصل الطلب إلى 157 متحكّمًا بلا سياق.

---

## 1. النمط الآمن

```php
// ✅ ينفّذ داخل سياق مستأجر ثم يستعيد ما كان — حتّى عند الاستثناء
$rows = TenantContext::withTenant($tenantId, fn () => Model::all());

// ✅ تجاوز مؤقّت لعمل إداري محدَّد، ويُغلق بعده حتمًا
$user = TenantContext::withBypass(fn () => User::find($id));

// ✅ عند الحاجة الحقيقية لتحكّم يدوي
$snap = TenantContext::snapshot();
try { /* … */ } finally { TenantContext::restore($snap); }
```

```php
// ❌ يمسح سياق المتصل بدل أن يستعيده
TenantContext::set($tenantId);
$rows = Model::all();
TenantContext::reset();

// ❌ «آمن» ظاهريًّا: finally يمسح ولا يستعيد
TenantContext::set($tenantId);
try { $rows = Model::all(); } finally { TenantContext::reset(); }
```

---

## 2. لماذا `reset()` اليدوي خطر

`reset()` **لا يُعيد** السياق السابق — بل يُفرغ كل شيء: المستأجر والمؤسسة
وورشة العمل و`bypass`. و`set($tenantId)` بوسيط واحد **يكتب الحقول الثلاثة**،
فيمسح المؤسسة ضمنًا.

النتيجة ليست خطأً بل **صمتًا**:

| ما وقع فعلًا | الأثر |
|---|---|
| استعلام بعد `reset()` | يعود فارغًا → يُقرأ «لا سجلّ» فيُتخطّى حارس تكرار |
| بحث عن مستقبِل بعد `transition()` | يعود فارغًا → **يسقط الإشعار بصمت** (وقع في العقود والمحتوى) |
| `set($tenantId)` قبل `authorize()` | تُمسح المؤسسة → `roleIn($orgId)` فارغ → **403 للجميع** |
| `bypass(true)` بلا إغلاق مضمون | يبقى مفتوحًا في عامل الطوابير → **قراءة وكتابة بلا نطاق** |
| `reset()` في وسيط بعد ضبط السياق | الطلب كلّه بلا سياق → 157 متحكّمًا يُعوّضون يدويًّا |

**العزل fail-closed**: النتيجة الفارغة تعني «لا سياق» بقدر ما تعني «لا بيانات».
لا يجوز أن يُقرأ الفراغ على أنه غياب سجلّات دون التأكّد من السياق.

## 3. العدد قبل وبعد

| المرحلة | الاستدعاءات اليدوية |
|---|---|
| قبل المسح | **370** |
| بعد إصلاح الوسائط وحذف التعويض | 275 |
| بعد تحويل الخدمات والأفعال | 235 |
| بعد المتحكّمات | 93 |
| بعد النطاق (`app/Domain`) | 32 |
| **الآن** | **صفر** |

الصفر يعني: لا `set` ولا `reset` ولا `bypass` يدويًّا في `app/` كلّه — عدا ستّة
ملفّات مذكورة بالاسم أدناه.

### المستثنيات النهائية — بالاسم والسبب

| الموضع | السبب |
|---|---|
| `Domain/Tenancy/Support/TenantContext.php` | يُعرّف النمط الآمن نفسه؛ `withTenant`/`withBypass` مبنيّتان فوقه |
| `Domain/Tenancy/Support/SetTenantContext.php` | Bootstrap داخلي: يبني سياق الطلب من المستخدم |
| `Http/Middleware/SetTenantContext.php` | وسيط الطلب الأساسي — نقطة الدخول الوحيدة للسياق |
| `Http/Middleware/EnsureClientMember.php` | يُنشئ سياق بوابة العميل لمدى الطلب |
| `Http/Middleware/EnsureCreator.php` | كسابقه، لبوابة صانع المحتوى |
| `Http/Middleware/EnsurePartnerMember.php` | كسابقه، لبوابة الشريك |

قيدٌ على الوسائط الثلاثة: الاستثناء مقصور على **إنشاء** السياق — استدعاء `set`
واحد قرب نهايتها. وسيطٌ يضبط ثم يُعيد داخل نفسه يعود إلى العيب الأصلي، وهو ما
وقع فعلًا وكلّف 157 تعويضًا في المتحكّمات.

> `EnsureAgencyMember` **ليس** مستثنًى: يقرأ السياق ويتحقّق منه ولا يضبطه.

### ما وجده المسح — لا تحويل تجميلي

| الموضع | ما كان |
|---|---|
| `CreateTenantNoteJob` (مثال المهامّ المرجعي) | `reset()` ثم `set($prev)` — يستعيد المستأجر ويُسقط المؤسسة وورشة العمل والتجاوز |
| `NotificationService` | نسخة **خاصّة** من `withTenant` تُظلّل الحقيقية؛ إن كان المتصل متجاوِزًا أعادت التجاوز وحده |
| `CreatorCapabilityService` | تأخذ لقطة في `$__ctx` **ولا تستعيدها أبدًا**؛ تُعيد ثلاثة حقول بيدها |
| `ConvertPublisherToInfluencer` | ثلاثة `reset()` على ثلاثة مخارج، ولا رابع لمسار الاستثناء |
| `TeamController` | `finally { reset() }` يعمل بعد `return` — فيُبنى الردّ بلا سياق |
| `PlatformController` | تجاوز إداري عبر كل المستأجرين يُغلق بـ`reset()`؛ استثناء يتركه مفتوحًا |
| ثلاثة متحكّمات بوابات | `set()` ثم `try` بـ`finally` **فارغ** — ضبطٌ دائم بلا استعادة |
| `SeedE2e` · `SeedPreviewAccounts` | `reset(); set($t, $org);` في سطر واحد — النمط الممنوع حرفيًّا |
| `DeliverableMatchController` · `SeedE2e` | تعويضٌ عن مسحٍ لم تعد الخدمة تفعله — والتعليق يبرّره بماضٍ انقضى |

## 4. الأنماط المعتمَدة لكل سطح

| السطح | النمط |
|---|---|
| HTTP | الوسيط يُنشئ السياق مرّة؛ المتحكّم **لا يضبط شيئًا** ولا يُعيد |
| الخدمات/الأفعال | `withTenant($model->tenant_id, fn () => …)` وتُعيد قيمتها |
| المهامّ | `tenant_id` صريح في الحمولة ← `withTenant` داخل `handle()` |
| Webhooks | تحقّق التوقيع ← استخرج المستأجر بـ`withBypass` ← عالج بـ`withTenant`؛ وبلا مستأجر يُرفض |
| الأوامر/المجدوِل | مرور صريح على المستأجرين، كلٌّ داخل `withTenant` |
| الإداري عبر المستأجرين | `withBypass` لعمل محدَّد فقط، ويُغلق بعده |

## 5. المهامّ والطوابير

المهمّة تعمل **بلا وسيط**، فلا أحد يضبط لها السياق. القاعدة:

```php
class SyncStoreOrdersJob implements ShouldQueue
{
    // ✅ المستأجر صريح في حمولة المهمّة، لا مستنتَج
    public function __construct(public int $tenantId, public int $storeId) {}

    public function handle(): void
    {
        TenantContext::withTenant($this->tenantId, function () {
            // كل الاستعلامات هنا مُنطّقة، والسياق يُغلق حتمًا بعدها
        });
    }
}
```

خطر خاصّ بالعامل طويل العمر: العملية الواحدة تنفّذ مهامّ متتابعة. تجاوز بقي
مفتوحًا في مهمّة **يمتدّ إلى التالية**. وقع فعليًّا في `FinalizeCreatorFilesJob`
قبل تحويله.

## 6. الـWebhooks

الطلب الوارد من مزوّد خارجي **لا هوية له**. القاعدة:

1. تحقّق من التوقيع أوّلًا.
2. استخرج المستأجر من سجلّ الاتصال (بـ`withBypass` مرّة واحدة).
3. نفّذ بقيّة المعالجة داخل `withTenant`.
4. Webhook بلا مستأجر معروف **يُرفض** — لا يُعالَج بلا نطاق.

```php
$connection = TenantContext::withBypass(
    fn () => StoreConnection::where('external_id', $payload['store_id'])->first()
);
abort_unless($connection, 404);

TenantContext::withTenant($connection->tenant_id, fn () => $this->ingest($payload));
```

## 7. السياق المتداخل

`withTenant` يدعم التداخل ويعود طبقةً طبقة. وإعادة تأكيد **المستأجر نفسه**
تُبقي المؤسسة وورشة العمل — وإلّا لَأسقط كودٌ داخليٌّ مؤسسةَ الطلب بمجرّد أن
يُعيد تأكيد مستأجره (وهو سبب أعطال 403).

```php
TenantContext::withTenant(5, function () {          // مستأجر 5 (+ مؤسسته محفوظة)
    TenantContext::withTenant(9, fn () => /* 9 */); // طبقة داخلية
    // عاد إلى 5 بمؤسسته
});
```

## 8. الحارس

```bash
./scripts/check-tenant-context-safety.sh    # يفشل عند أي استدعاء يدوي جديد
```

**صارم، بلا سقّاطة وبلا أساس.** `set/reset/bypass` ممنوعة في `app/` كلّه، عدا
الملفّات الستّة المذكورة في §3 بالاسم والسبب.

كان الحارس سقّاطة تقبل الدَّين القائم وتمنع الزيادة — وهو الصواب حين كانت
الاستدعاءات بالمئات. لكن السقّاطة تقول «ما كان موجودًا مقبول»، وذلك ختمٌ على
الوضع القائم لا حراسة. ببلوغ الصفر صار الأساس الواسع بلا معنى، فحُذف
`scripts/tenant-context-baseline.txt`.

**والحارس نفسه محروس.** كان معطوبًا عند النجاح: ببلوغ الصفر يخرج `grep` بـ1
فيقتله `set -e` بلا رسالة — سكربتٌ أخضر دائمًا يبدو حراسةً وهو عمى.

> لا يوجد CI في هذا المستودع بعد. الإنفاذ يجري عبر مجموعة الاختبارات، وهي ما
> يُشغَّل فعلًا. إن أُضيف CI فليستدعِ السكربت مباشرةً أيضًا.

## 9. الاختبارات الحارسة

`tests/Feature/TenantContextSafetyTest.php` — الأدوات:

| ما يحرسه | لماذا |
|---|---|
| استعادة بعد استثناء · بعد `abort` · بعد `return` مبكّر | الخروج من النطاق له طرق كثيرة، وكلّها يجب أن تستعيد |
| استعادة بعد استثناء من طبقة **متداخلة** | كلّ طبقة تستعيد لقطتها، فلا تبقى حالة وسطى |
| القيمة تعبر النطاق | البديل هو تسريب متغيّر من داخل الكتلة — وهو ما كسر التحويل الآلي أربع مرّات |
| حفظ المؤسسة وورشة العمل عند إعادة تأكيد المستأجر | إسقاطهما هو سبب أعطال 403 |
| حفظ التجاوز واستعادته لا فرضه | تجاوز يبقى مفتوحًا = قراءة وكتابة بلا نطاق |
| أوامر متتابعة في عملية واحدة | العملية طويلة العمر تجعل التسريب يمتدّ إلى ما بعدها |

`tests/Feature/TenantIsolationSweepTest.php` — الأسطح: HTTP · مهامّ · عامل
طوابير متتابع · Webhooks · مجدوِل · ذاكرة · إشعار بعد انتقال · مدير النظام
للقراءة فقط.

`tests/Feature/TenantContextGuardTest.php` — الحارس: يزرع استدعاءً يدويًّا في
سبع طبقات ويتأكّد أنه يرفض كلًّا منها بخروج 1، ويتأكّد أن قائمة الاستثناءات لم
تتّسع ولم يعد الأساس الواسع.

اختبار واحد يوثّق العيب نفسه عمدًا: استعلام بلا سياق يعود **فارغًا لا خطأً**.
الصمت هو العطل.

## 10. درسٌ في التحويل: الآليّ لا يكفي

أربعة أنماط كسرت التحويل الآلي، ولم يكشفها إلا تشغيل الاختبارات:

1. **إغلاق فقد `use (...)`** لمتغيّر يستعمله. أخطرها `ImportLegacyClients`:
   سقط `$actor` فصار null، فرمى `CreateClient`، فابتلعه `catch (\Throwable)`
   وعُدّ كل سطر «متخطًّى». **صفر عميل مستورَد، وصفر رسالة خطأ.**
2. **متغيّر يُسنَد داخل الكتلة ويُستعمل بعدها** ⇒ يلزم `return` من الإغلاق
   والتقاط ناتجه.
3. **ترتيب وسائط `withTenant($tenantId, $fn, $orgId)`** — الآليّ وضع `$orgId`
   في موضع الإغلاق.
4. **تعويضٌ عن عيبٍ زال.** بعض المواضع كانت تُعيد ضبط السياق بعد نداء خدمة،
   وتعليقٌ يبرّر ذلك بأن الخدمة «تمسح السياق داخليًّا» — ولم تعد. التعويض بقي
   يعمل ضدّ سياقٍ سليم. اقرأ ما تستدعيه، لا ما يقوله التعليق عنه.

القاعدة المستخلَصة: المحوِّل الآليّ يرفض كل كتلة يعبرها متغيّر ويطلب اليد؛
ويُفحَص بعده كل إغلاق أُنشئ بحثًا عن متغيّر مستعمَل غير ملتقَط.
