# Experience Audit — InfluencerHub

تدقيق التجربة: أين نقف اليوم من تحويل CRUD → مساحات عمل تفاعلية، وما التالي.

## 1. المنجز (Foundation)

| العنصر | الحالة |
|-------|-------|
| نظام تنقّل مركزي + مصطلحات موحّدة | ✅ مطبّق عبر الوكالة/العميل/الشريك |
| لوحة الأوامر ⌘K | ✅ عبر البوابات الثلاث، مُختبرة E2E |
| مكوّنات مساحة العمل (header/summary/status-badge) | ✅ في Design System، جاهزة للتطبيق |
| واجهة موجّهة بالإجراء (لا قوائم status يدوية) | ✅ في وحدات المراجعة/الطلبات/المحتوى (آلات حالة بالأحداث) |
| حالات فارغة ذات معنى | ✅ جزئيًا (لوحات العميل/الوكالة) |

## 2. فجوات التطبيق (الأولوية التالية)

| الصفحة | الفجوة | الإجراء المقترح |
|-------|-------|-----------------|
| تفاصيل الحملة | رأس عادي بلا summary strip | تطبيق `<x-workspace-header>` + `<x-summary-strip>` (مخرجات/معتمد/متبقّي) |
| تفاصيل طلب الخدمة | لا مخطط زمني لسير العمل ظاهر | إضافة workflow timeline من status history |
| تفاصيل العميل | تبويبات موجودة، ينقص رأس مساحة العمل الموحّد | تبديل الرأس إلى `<x-workspace-header>` |
| نماذج الإنشاء الطويلة | كل الحقول دفعة واحدة | Progressive Disclosure + Quick Create (راجع FORM-REDUCTION-PLAN) |

## 3. منهجية التطبيق

لكل صفحة تفاصيل:
1. استبدل الرأس اليدوي بـ `<x-workspace-header>` (عنوان + `<x-status-badge>` + حقائق + أزرار إجراء سياقية).
2. أضِف `<x-summary-strip>` بمؤشرات حقيقية من PostgreSQL.
3. حوّل أي قائمة status منسدلة إلى أزرار إجراء موجّهة.
4. تحقّق في المتصفّح + اختبار خلفي + Commit.

## 4. مبادئ حاكمة

- **لا بيانات وهمية** — كل مؤشر من PostgreSQL.
- **PostgreSQL مصدر الحقيقة** — لا localStorage للبيانات التشغيلية.
- **لوحة الأوامر للتنقّل فقط** — لا تُسرّب بيانات تشغيلية للعميل.
